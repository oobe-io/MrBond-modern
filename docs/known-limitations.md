# 既知の制約

MrBond-modern が Mr.Bond 完全パリティに対して持つ残課題と、そのスコープ判断。

## 1. ULP ドリフト（ARM64 Mac vs x86 Windows）

**症状:** SN_valve, arie_jet_ver2 等のモデルで CSV 出力が Mr.Bond リファレンス
と最終桁 1 ULP 単位でズレる（例: `1.212127e-09` vs `1.212126e-09`）。

**原因:** プラットフォーム間の IEEE 754 浮動小数実装差:
- FMA（Fused Multiply-Add）の利用有無
- `sin/cos/sqrt` 等の超越関数の内部アルゴリズム
- コンパイラの `-ffast-math` 系最適化
- libm の実装差（gcc, clang, MSVC, Borland bcc32）

**大きさ:** 絶対誤差 ~2⁻⁵² × |value| ≈ 1e-16 × |value|。物理計測ノイズより遥かに小さい。

**対応方針:** **Won't fix**。1 ULP 以下の差異は数値シミュレーションでは「同等」
とみなす業界標準。完全バイト一致には下記のいずれかが必要:
- JS の `Math` を software-float で置換（200-500倍遅くなる）
- Mr.Bond と同じ ISA （x86）で実行
- 全操作を整数演算で再実装

デモ・実用上は無視可能。必要なら呼び出し側で近似比較を使う。

## 2. Mr.Bond 純正 BGE ファイルの完全パース

**症状:** うちの `writeBge()` で生成したBGEは完全 round-trip するが、Mr.Bond
本体が生成した実 BGE ファイルは要素 5 付近（offset 1651 等）でパース失敗。

**原因:**
- ジャンクション要素のパッドアトム数が微妙に違う（うち: 23、実: 25〜26）
- 一部要素で subtype アトムが欠落し、`grid_w` が直接出ている

**対応方針:** **部分対応**。ヘッダ（要素数・ボンド数）と基本形は取れる。
完全対応には原作 Mr.Bond の `BGE_R.C`（リーダー側コード）を取得するか、
適応パースロジックをさらに追加する必要がある。

ワークアラウンド: うちのエディタで新規作成 → BGE 書き出し → Mr.Bond で開く
（writer 側レイアウトが Mr.Bond に合えば OK）ルートは round-trip 成立。

## 3. Mr.Bond C 生成コードの ARM64 Mac でのクラッシュ

**症状:** 20170202_FULLmodel, arie-sen-v_1, arie_jet_3, プランジャーポンプ,
絞りマックス の5モデルが原作 C コードをコンパイル・実行すると SIGKILL。

**原因:** Mr.Bond の C コード側バグ（変数シャドウ、PT 配列のバッファオーバー
フロー、未初期化値など）が ARM64 Mac の stack canary で検出される。bcc32
（Borland C++）は寛容で通していた。

**対応方針:** **Mr.Bond 側の問題**。うちの TypeScript 実装はこれらのバグを
回避しているので、これらのモデルでもうち側は動作する可能性が高い（要検証）。

## 4. BG_改良_FB無限ループ回避モデル の発散

**症状:** モデル名通りフィードバック制御系で、シミュレーションが発散しがち。

**原因:** モデル設計上の特性（無限ループ回避のための条件分岐）。

**対応方針:** **モデル側の特性**。うちのソルバは正しく動作しており、発散
するのはモデルの物理挙動。

## 5. 双方向反復（微分因果・代数ループ）

**症状:** `ND > 0` のモデル（うちが回収した15モデル全部 ND=0）で SOLV が
呼ばれる。実装は Runge.f を忠実に移植済みだが、ND>0 のテストデータがない。

**対応方針:** 実装済み、動作は Runge.f と等価なはず。将来 ND>0 モデルが
出てきた時点で検証。
